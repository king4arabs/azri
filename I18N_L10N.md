# I18N_L10N

> Internationalization and localization strategy for AZRI. Arabic-first, RTL-first, then English/LTR, then GCC dialects/global.

## Principles

1. **Arabic-first, not Arabic-translated.** Source copy is reviewed in Arabic by a native KSA speaker; English mirrors it.
2. **RTL is a first-class layout, not a flip.** Designs are reviewed in both directions before merge.
3. **No hard-coded strings.** Everything user-visible goes through the i18n catalog.
4. **No string concatenation** to build sentences. Use ICU MessageFormat with placeholders, plurals, gender, select.
5. **Locale negotiation is explicit.** Accept-Language → user preference → fallback chain (`ar-SA → ar → en`).
6. **Region ≠ language.** A user in KSA may prefer English; respect that.
7. **Numerals & dates respect locale.** Hijri/Gregorian toggle; Arabic-Indic vs Western digits user preference.

## Supported locales

| Locale | Status (target) | Notes |
| --- | --- | --- |
| `ar-SA` | Primary, v0.2.0 | Source of truth for review |
| `en` | Primary, v0.2.0 | Mirror translation |
| `ar-AE`, `ar-BH`, `ar-KW`, `ar-OM`, `ar-QA` | v0.8.0 | GCC dialect refinements where needed |
| Other | TBD | Driven by demand |

## Tooling (target)

- **Web:** `next-intl` (App Router-native) or `react-intl`. ICU MessageFormat. Catalogs in JSON.
- **iOS:** Native `String Catalogs` with stringsdict for plurals.
- **Pseudo-localization** in dev to surface untranslated strings (`[!! Übêrländé string here !!]`).
- **Translation management:** Crowdin or Lokalise (decide ahead of v0.3.0). Source-locale review by native speaker.

## RTL specifics

- Use **logical properties** in CSS: `margin-inline-start/end`, `padding-block-*`, `border-inline-*`, `inset-inline-*`.
- Use logical text alignment (`text-align: start/end`).
- Mirror direction-sensitive icons (chevrons, back arrow) only when semantically appropriate.
- Never mirror logos, numerals, code blocks, or media controls that have universal direction.
- Test all components rendered in `dir="rtl"` *and* `dir="ltr"`; both must look right.

## Number, date, currency

- `Intl.NumberFormat` and `Intl.DateTimeFormat` with the active locale.
- User preference toggle for **Hijri / Gregorian** display where relevant (medication schedules, episode dates).
- Currency formatting respects locale: `SAR` shows as "ر.س." or "SAR" depending on user preference.

## Healthcare-specific localization

- Medical terminology reviewed by an Arabic-speaking clinician; we maintain a glossary.
- Plain-language standards: target a reading level appropriate for adult patients (avoid academic medical jargon in patient-facing surfaces).
- Disclaimers translated and reviewed; never machine-translated.

## Workflow

1. PR introduces a string → goes into the source catalog (Arabic primary).
2. CI fails if any visible string is untranslated in the supported locales.
3. Native-speaker review required for any patient- or clinician-facing copy change (PR template gate).
4. Pseudo-localization run weekly on `main` to catch leaks.
